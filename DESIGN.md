# GCP SDK IAM Permission Detector — Design Document

**Status:** Draft
**Last Updated:** 2026-03-13

---

## 1. Problem

Developers using GCP client libraries need to know which IAM permissions their code requires. Today this requires manually cross-referencing SDK method names against scattered GCP documentation. There is no unified, machine-readable mapping from Python SDK method to IAM permission(s).

## 2. Goals

- Automatically discover all installed `google-cloud-*` Python SDK packages
- For every public method on every Client class, produce a static mapping to IAM permission(s)
- Use the **Google APIs Discovery Service** and **Gemini 3.0** to generate mappings at build time
- Ship as a flat lookup file (`iam_permissions.json`) — zero network calls, zero LLM inference at runtime
- Provide a clean `PermissionResolver` interface decoupling detection from mapping source

## 4. Key Interfaces

### PermissionResolver

```python
class PermissionResolver(ABC):
    @abstractmethod
    def resolve(self, service_id: str, class_name: str, method_name: str) -> PermissionResult | None: ...

@dataclass(frozen=True)
class PermissionResult:
    permissions: list[str]              # e.g. ["bigquery.jobs.create"]
    conditional_permissions: list[str]  # e.g. ["storage.objects.delete"]
    is_local_helper: bool               # True for path builders, constructors
    notes: str = ""
```

`StaticPermissionResolver` loads `iam_permissions.json` for O(1) lookups. Key format: `{service_id}.{class_name}.{method_name}`, with `*` wildcard for class.

### MethodSig

```python
@dataclass(frozen=True)
class MethodSig:
    min_args: int
    max_args: int
    has_var_kwargs: bool
    class_name: str
    service_id: str      # e.g. "bigquery"
    display_name: str    # e.g. "BigQuery"
```

Matching: method name in `MethodDB` (dict lookup), then arg count within `[min_args, max_args]`.

## 5. Runtime Performance Constraint

**The scanner must not import GCP SDK packages at runtime.**

Measured: `build_method_db()` imports 63 packages and introspects 14K signatures — takes 13.4 seconds. Actual file scanning takes 17ms for 25 files. Startup is 99.7% of wall time.

The fix: `method_db.json` is a pre-built build artifact, loaded at runtime like `iam_permissions.json`. The runtime path is:

```
Load 3 JSON files (~1ms) → scan files (~0.05ms/file)
```

Build-time introspection stays in `introspect.py` but is only called by build scripts, never by the scanner or CLI.

## 6. Static Build Artifacts

Three JSON files checked into the repo. All generated by build scripts, loaded at runtime:

| File | Generated by | Contents |
|---|---|---|
| `service_registry.json` | `build_service_registry.py` + `fix_registry_metadata.py` | 62 services, modules, IAM prefixes |
| `iam_permissions.json` | `build_permission_mapping.py` | method → IAM permission mappings |
| `method_db.json` | `build_method_db.py` | pre-built method signatures for matching |
| `iam_role_permissions.json` | `gcloud iam roles list` | 12,879 valid IAM permissions (ground truth for validation) |

## 9. Migration from Current Design

| Current | New |
|---|---|
| `iam_perms.py` — 400+ lines of `_add()` calls + pattern rules | `iam_permissions.json` — flat lookup, no code |
| `_LABEL_OVERRIDES` dict for stable naming | `service_registry.json` is the single source of truth |
| Pattern rules for `*_path`, `get_iam_policy`, etc. | Gemini maps all methods uniformly |
| `lookup_permissions(service, method, class)` by display name | `resolver.resolve(service_id, class_name, method)` by service_id |
| Manual curation for each new service | Automated: install package → re-run build pipeline |

Existing curated mappings in `iam_perms.py` become the ground truth test suite in `tests/test_known_mappings.py`.

## 10. Open Questions

1. ~~**Async client duplication**~~ **RESOLVED**: AsyncClient skipped during build. Same methods, same permissions. Halves the method DB.
2. ~~**Runtime startup**~~ **RESOLVED**: Measured 13.4s startup (SDK introspection) vs 17ms scan. Pre-build `method_db.json` at build time, load at runtime. See §5.
3. **Version pinning**: pin discovery docs to specific versions to avoid mapping drift?
4. **Coverage threshold**: 100% of non-private methods mapped or marked `local_helper`?
4. ~~**False positive mitigation**: file-level check for GCP imports before reporting?~~ **RESOLVED**: Scanner requires GCP imports. No imports = no findings. Uses tree-sitter AST to extract imports, with fast `"google.cloud" in source` early exit.

### Build Pipeline

5. **Observability**: Build scripts need progress bars, ETAs, and structured logging. Currently just print to stderr with no indication of total work remaining. Should show: `[service 3/62] Cloud Storage: batch 2/3 (50 methods mapped, 0 errors)`.
6. **Resumability**: If the Gemini mapping job fails mid-run (API error, rate limit), there's no way to resume from where it left off. Should checkpoint progress and support `--resume`.
7. **Caching**: Discovery docs and IAM reference pages should be cached locally with TTL. Re-fetching on every build is wasteful.
8. **Parallelism**: Gemini API calls are sequential with 0.5s delay. Could pipeline requests or use async to overlap I/O wait with processing.
9. **Validation feedback loop**: When Gemini produces a mapping that fails validation (hallucinated permission), should we retry with feedback, or just flag for manual review?

### Registry

10. **`service_id` != `iam_prefix` gotcha**: Documented in [docs/service-registry.md](docs/service-registry.md). Gemini-assisted fix via `fix_registry_metadata.py`. But should we validate this in CI? A test that checks every `iam_prefix` in the registry against known GCP IAM permission prefixes.
11. **New service discovery**: When a user installs a new `google-cloud-*` package not in the registry, the scanner silently ignores it. Should we warn? Auto-add with defaults?

### Testing

12. **Real-world repo testing**: Test against actual GCP codebases (e.g. `GoogleCloudPlatform/python-docs-samples`) to validate precision and recall. Should be `@pytest.mark.slow` integration tests.
