# Permission Manifest

A YAML artifact generated by `iamspy scan --manifest` that lists the GCP
permissions and APIs a codebase requires.

## Generating a manifest

```bash
iamspy scan --manifest iam-manifest.yaml src/
```

Add `--trace` to include source file and line number for each permission:

```bash
iamspy scan --manifest iam-manifest.yaml --trace src/
```

## Manifest format

```yaml
version: "1"
generated_by: "iamspy scan src/"
generated_at: "2026-03-18T03:36:31Z"

services:
  enable:
    - bigquery.googleapis.com
    - storage.googleapis.com
    - secretmanager.googleapis.com

permissions:
  required:
    - bigquery.jobs.create
    - secretmanager.versions.access
    - storage.buckets.get
    - storage.objects.create
  conditional:
    - bigquery.tables.create
    - storage.objects.delete
```

With `--trace`, a `sources` section maps each permission to its call sites:

```yaml
sources:
  bigquery.jobs.create:
    - file: src/etl.py
      line: 42
      method: query
  storage.objects.create:
    - file: src/upload.py
      line: 15
      method: upload_from_filename
```

## Sections

### `services.enable`

GCP APIs that must be enabled in the project. Derived from the `api_service`
field in the service registry (e.g. `bigquery.googleapis.com`). Enable with:

```bash
gcloud services enable bigquery.googleapis.com --project=PROJECT
```

### `permissions.required`

IAM permissions unconditionally required by the scanned code. Flat list,
deduplicated and sorted.

### `permissions.conditional`

Permissions required only in certain runtime conditions (e.g. "only if
overwriting an existing object"). Flat list of permission strings.

### `sources` (optional, `--trace` only)

Maps each permission to the source locations that require it. Keyed by
permission name, each entry is a list of `{file, line, method}` objects.

## Using the manifest

**Check it into source control.** When a PR adds a new GCP call that requires
a new permission, the manifest diff shows it in code review.

**CI gate.** Regenerate the manifest in CI and diff against the checked-in
version. If they differ, the PR adds permissions that haven't been reviewed.

```yaml
# .github/workflows/iam-check.yml
- run: |
    iamspy scan --manifest /tmp/manifest.yaml src/
    diff iam-manifest.yaml /tmp/manifest.yaml
```

**Provision manually.** Read the manifest and run the corresponding `gcloud`
commands. The manifest tells you exactly what to enable and grant.

## Not yet implemented

These features are planned but do not exist in the current version:

- **`roles` section** — suggested predefined roles and custom role YAML
  generation. Requires a set-cover algorithm over `iam_role_permissions.json`.
- **`project` field** — target project for provisioning.
- **`iamspy apply`** — automated provisioning via `gcloud` subprocess.
- **`iamspy diff`** — detect drift between manifest and source code.
- **Conditional `reason` field** — structured entries with human-readable
  reasons (currently conditional permissions are flat strings).
- **Bindings section** — which service account gets which role.

These are tracked in `docs/todo.md`.
