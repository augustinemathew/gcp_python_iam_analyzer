# Permission Manifest

A YAML artifact generated by `iamspy scan` that fully describes the GCP permissions, roles, and APIs a codebase requires. The manifest is checked into source control and drives automated permission provisioning via `iamspy apply`.

## Motivation

`iamspy scan` today produces human-readable output or a flat JSON array of findings. Neither is actionable: you still have to manually translate findings into `gcloud` commands. The manifest closes that gap.

The manifest is the single source of truth for what a service needs. It is:
- **Generated** from code by `iamspy scan --manifest`
- **Checked in** alongside the code that requires the permissions
- **Applied** via `iamspy apply` to provision resources in GCP
- **Diffed** in CI to detect when new code adds permissions not yet provisioned

---

## Manifest Format

```yaml
version: "1"
generated_by: "iamspy scan src/"
generated_at: "2026-03-17T14:00:00Z"

# Optional. If set, apply uses this project unless overridden by --project.
project: my-gcp-project

services:
  enable:
    - bigquery.googleapis.com
    - storage.googleapis.com
    - cloudkms.googleapis.com

permissions:
  required:
    - bigquery.jobs.create
    - bigquery.tables.getData
    - storage.objects.get
    - cloudkms.cryptoKeyVersions.useToDecrypt

  conditional:
    - permission: storage.objects.delete
      reason: "only if overwriting an existing object"
    - permission: bigquery.datasets.create
      reason: "only if dataset does not already exist"

roles:
  # Suggested predefined roles (set-cover of required permissions).
  # These grant more than the minimum — use custom for least-privilege.
  suggested:
    - roles/bigquery.jobUser        # covers: bigquery.jobs.create
    - roles/bigquery.dataViewer     # covers: bigquery.tables.getData
    - roles/storage.objectViewer    # covers: storage.objects.get
    - roles/cloudkms.cryptoKeyDecrypterViewer

  # Custom role YAML. Exactly minimal — only the permissions this code needs.
  # Apply creates or updates this role in the project.
  custom:
    title: my-app-permissions
    description: "Minimum permissions for my-app (generated by iamspy)"
    stage: GA
    included_permissions:
      - bigquery.jobs.create
      - bigquery.tables.getData
      - storage.objects.get
      - cloudkms.cryptoKeyVersions.useToDecrypt

# Source traceability. Only present when scan is run with --trace.
sources:
  - file: src/etl.py
    line: 42
    method: create_job
    permissions: [bigquery.jobs.create]
  - file: src/etl.py
    line: 78
    method: list_rows
    permissions: [bigquery.tables.getData]
  - file: src/storage_client.py
    line: 15
    method: get_blob
    permissions: [storage.objects.get]
    conditional: [storage.objects.delete]
```

## Sections

### `services.enable`

GCP APIs that must be enabled in the project for the code to function. `ManifestGenerator` collects the `service_id` of every matched `MethodSig` from the scan results, looks each one up in the `ServiceRegistry`, and reads `entry.api_service`. An empty `api_service` is an error — manifest generation raises rather than silently omitting a service.

Applied via:
```
gcloud services enable bigquery.googleapis.com --project=PROJECT
```

### `permissions.required`

IAM permissions unconditionally required by the scanned code. Flat list, deduplicated across all findings. Permissions are in `resource.noun.verb` format (e.g. `bigquery.jobs.create`).

### `permissions.conditional`

Permissions required only in certain runtime conditions. Each entry includes the permission string and a human-readable `reason` derived from `PermissionResult.notes`. Always included in the manifest — no flag required.

### `roles`

Two role representations. Choose one or include both:

**`roles.suggested`** — predefined GCP roles that collectively cover `permissions.required`. Computed as a set-cover using `iam_role_permissions.json`. Each role has a comment showing which required permissions it provides. These grant broader access than strictly needed; useful for simple deployments or development environments.

**`roles.custom`** — a minimal custom IAM role containing exactly the required permissions. Follows the format accepted by `gcloud iam roles create --file`. Suitable for production least-privilege setups.

### `sources`

Optional traceability from each permission back to the line of code that requires it. Maps permissions to `{file, line, method}`. Useful for audits ("why do we need `cloudkms.cryptoKeyVersions.useToDecrypt`?"). Only present when `--trace` is passed to `scan --manifest`.

## CLI

### Generating a manifest

New flag on the existing `scan` command:

```bash
# Print manifest to stdout
iamspy scan --manifest src/

# Write to file
iamspy scan --manifest -o iam-manifest.yaml src/

# Include source traceability (sources section)
iamspy scan --manifest --trace -o iam-manifest.yaml src/
```

`--manifest` is an additional output mode. The existing terminal, `--json`, and `--compact` outputs are completely unchanged — `--manifest` does not replace them.

### Applying a manifest

New `apply` subcommand:

```bash
# Preview — print gcloud commands, do not execute
iamspy apply iam-manifest.yaml --dry-run

# Execute — run gcloud commands against the project in the manifest
iamspy apply iam-manifest.yaml

# Override project
iamspy apply iam-manifest.yaml --project my-other-project

# Apply only specific sections
iamspy apply iam-manifest.yaml --only services
iamspy apply iam-manifest.yaml --only roles.custom
iamspy apply iam-manifest.yaml --only permissions   # binds custom role to a member
```

`--dry-run` output example:
```
gcloud services enable bigquery.googleapis.com --project=my-gcp-project
gcloud services enable storage.googleapis.com --project=my-gcp-project
gcloud iam roles create my-app-permissions --project=my-gcp-project \
  --file=<(iamspy apply iam-manifest.yaml --emit-role-yaml)
```

`apply` requires `gcloud` to be on PATH and authenticated. It does not manage IAM bindings (member→role) unless a `bindings` section is added (see Open Questions).

### Detecting drift

New `diff` subcommand:

```bash
# Show permissions in src/ not covered by manifest
iamspy diff iam-manifest.yaml src/

# Exit 1 if drift detected (for CI)
iamspy diff --fail-on-drift iam-manifest.yaml src/
```

Useful as a pre-commit hook or CI check to enforce that new GCP SDK calls are reflected in the manifest before merging.

## Workflow

**Day 0 — initial setup:**
```bash
iamspy scan --manifest -o iam-manifest.yaml src/
# review and edit the manifest
git add iam-manifest.yaml
iamspy apply iam-manifest.yaml --dry-run   # review commands
iamspy apply iam-manifest.yaml             # provision
```

**Ongoing — adding new GCP calls:**
```bash
# After writing new code
iamspy diff iam-manifest.yaml src/         # see what's new
iamspy scan --manifest -o iam-manifest.yaml src/   # regenerate
git diff iam-manifest.yaml                 # review changes
iamspy apply iam-manifest.yaml             # apply delta
```

**CI gate:**
```yaml
# .github/workflows/iam-check.yml
- run: iamspy diff --fail-on-drift iam-manifest.yaml src/
```

## Design Decisions

### `googleapis.com` identifiers

`ServiceEntry` has `api_service: str` as a required first-class field — the exact name used by `gcloud services enable` (e.g. `bigquery.googleapis.com`, `cloudkms.googleapis.com`). This is populated at build time (s02) from the discovery doc or Cloud API catalog.

Heuristic derivation from `service_id` or `iam_prefix` is not used. Many services diverge: Compute Engine is `compute.googleapis.com` (not `gce.googleapis.com`), Resource Manager is `cloudresourcemanager.googleapis.com`. The correct value is always sourced explicitly. Any entry with an empty `api_service` emits a build-time warning and is omitted from `services.enable`.

### Flat permissions list vs. grouped by service

The manifest uses a flat list under `permissions.required`. Alternative: group by service.

```yaml
# Grouped (not chosen)
permissions:
  bigquery:
    - bigquery.jobs.create
    - bigquery.tables.getData
  storage:
    - storage.objects.get
```

Flat is simpler for tooling (set operations, diffing), and grouping is already handled by the `sources` section for traceability. The flat list is what `gcloud iam roles create` and custom role YAML expect anyway.

### Roles: suggested vs. custom

Both are included in the manifest by default. They serve different consumers:
- **Suggested predefined roles** — dev environments, quick start, when least-privilege isn't required
- **Custom role** — production, security-sensitive workloads, compliance requirements

`iamspy apply --only roles.custom` applies only the custom role. The operator can delete the `suggested` section if they don't want it.

### Conditional permissions

Conditional permissions always appear in the manifest under `permissions.conditional` with human-readable reasons. No flag required. They are kept separate from `required` to avoid over-provisioning — the custom role includes only `required` permissions. The operator reviews `conditional` and decides whether to add any to the custom role manually.

### Traceability

Included by default. `--no-trace` omits the `sources` section for a leaner manifest. The traceability data is the same as `--json` output, just embedded in the manifest structure.

## Open Questions

1. **Bindings section** — should the manifest include `bindings` (which service account gets the role)? This makes `apply` able to do end-to-end provisioning but ties the manifest to a specific identity. Probably phase 2: start with roles + services, add bindings later.

   ```yaml
   # Possible future bindings section
   bindings:
     - member: serviceAccount:my-app@my-project.iam.gserviceaccount.com
       role: projects/my-project/roles/my-app-permissions
   ```

2. **Role suggestion algorithm** — a greedy set-cover over `iam_role_permissions.json` minimizes the number of predefined roles. Worth implementing, or too complex for the value? Alternative: just list the per-service viewer/editor roles and let the operator prune.

3. **`apply` execution model** — call `gcloud` directly via subprocess, or emit a shell script the operator runs? Direct execution is more ergonomic but requires auth plumbing. Shell script is safer and easier to review. Start with `--dry-run` only, add execution later.

4. **Manifest updates are destructive** — `iamspy scan --manifest` regenerates the whole file. If the operator has hand-edited the manifest (e.g. added a `bindings` section), regeneration clobbers it. Options: merge strategy, preserve-unknown-sections, or explicit separation of generated vs. manual sections.

5. **`diff` granularity** — should `diff` report at the permission level (new permission X is not in the manifest) or the finding level (new call at `src/foo.py:42` is not covered)? Permission-level is more useful for the CI gate; finding-level is better for developer feedback.

## Implementation Plan

### Phase 1 — manifest generation

1. Add `api_service: str` field to `ServiceEntry` in `models.py`, populate in registry (see `docs/api-service-field.md`).
2. Add `ManifestGenerator` class that takes `list[ScanResult]` + `ServiceRegistry` and produces a manifest data structure. For `services.enable`: collect unique `service_id`s from matched findings → look up each in the registry → emit `entry.api_service`. Skip and warn on empty `api_service`.
3. Add `--manifest` and `--trace` flags to `cmd_scan`. Pass `registry` through to `ManifestGenerator`.
4. Serialize to YAML (add `pyyaml` or `ruamel.yaml` dependency).

### Phase 2 — apply (dry-run)

5. Add `apply` subcommand that reads the manifest and emits `gcloud` commands.
6. `--dry-run` only initially. Execution (subprocess) deferred.

### Phase 3 — diff

7. Add `diff` subcommand. Scans source, compares to manifest, reports new/removed permissions.
8. `--fail-on-drift` for CI.

### Phase 4 — role suggestion

9. Implement greedy set-cover against `iam_role_permissions.json` to populate `roles.suggested`.
10. Custom role YAML generation is straightforward (just `included_permissions` = `required`).
